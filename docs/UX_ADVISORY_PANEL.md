> 2026-04-29

---

## 0. 왜 외부 평가가 필요한가

자체 시각만으로는 평가 자체가 불가능한 영역이 있다.

- **문화적 정확성** — 한국사를 잘 아는 비평가가 아니면 "사극 미화" 같은 함정을 잡지 못한다
- **인지적 부하** — 만든 사람은 자기 앱을 절대로 처음 보는 사람처럼 쓸 수 없다
- **모바일 한 손** — 만든 사람은 시뮬레이터로 보고 PC에서 결정한다
- **접근성** — 스크린 리더로 써보지 않으면 못 본다
- **타 매체 비교 좌표** — Apple/Pudding/NHK를 실제로 만든 사람의 시각은 다르다

이 5축은 한 사람이 다 못 본다. **6명의 전문가 패널**이 필요하다.

## 2. 평가 방법론 (Methodology)

### 2.1 단계

```
Phase A. Pre-evaluation (1주)
├── 패널 6명 모집·계약
├── 의뢰자가 라이브 URL + 코드베이스 + 본 문서 + ROADMAP + DESIGN_BRIEF 공유
└── 각자 자율 사용 (5-10시간) — 첫인상 노트 작성

Phase B. Independent Evaluation (2주)
├── 각자 자기 영역 평가 (위 시간 견적)
├── 좌석 간 간섭 금지 (편향 방지)
└── 산출물 코디네이터 제출

Phase C. Synthesis Workshop (1일)
├── 6명 + 코디네이터 + 의뢰자 라운드테이블 (3-4시간)
├── 충돌 의견 토론
└── 우선순위 합의

Phase D. Final Report (1주)
├── 코디네이터 통합 보고서 작성
├── 우선순위 매트릭스
└── 의뢰자 발표 미팅 (1시간)
```

총 기간: **5주**

### 2.2 평가 대상 시나리오 (필수)

각 좌석은 다음 6개 시나리오를 반드시 통과시킨다.

1. **첫 30초 시나리오** — 처음 들어온 사용자가 "이건 뭐지?" 답을 30초에 얻는가
2. **사도세자 도달 시나리오** — 검색 없이 사도세자 사건 L3까지 가는 데 클릭 몇 번
3. **공유 시나리오** — 가장 인상 깊은 사건을 카톡으로 공유 가능한가
4. **다음 왕 시나리오** — 한 왕 다 봤을 때 자연스럽게 다음 왕으로 넘어가는가
5. **L4 한문 시나리오** — 한문 모르는 사용자가 L4에서 좌절하지 않는가
6. **모바일 지하철 시나리오** — 출근 지하철 한 손 환경에서 5분 머무를 수 있는가

### 2.3 결함 분류

각 결함은 다음 6필드로 기록한다:

```
ID:        E-001
좌석:      Mobile UX
심각도:    3 (0=정보, 4=치명)
위치:      L1 시대 카드 / 모바일 375px
현상:      시대 카드 그리드 무너짐, 가로 스크롤 발생
영향:      1차 페르소나 진입 30초 내 이탈 우려
권고:      세로 스택으로 변경, 시대 색상 좌측 4px 보더로 약화
수정비용: M (1-3일)
```

### 2.4 우선순위 매트릭스

```
              수정비용 낮음          수정비용 높음
심각도 높음   [즉시 수정]            [Phase 단위 계획]
심각도 낮음   [배치 정리 시]         [수정 안 함 명시]
```

## 4. 권장 진행 순서

### 본 의뢰자에게 추천하는 진행안

```
Now (현재)
├── ① 디자이너 의뢰 (DESIGN_BRIEF) 1차안 도착 대기
└── ② 자체 평가 우선 진행 (코드/콘텐츠 기준)

After 디자이너 1차안 도착
├── ③ 1차안 + 현재 라이브를 묶어 패널 평가 의뢰
├── ④ 평가 결과를 디자이너에게 피드백
└── ⑤ 디자이너 2차안 작업

Before 배포
└── ⑥ 패널 재평가 (간소판 — A11y + Mobile 2명만)
```

이유:
- 디자이너 1차안 없이 평가하면 "지금 약점"만 보고 "디자인 방향"을 못 봄
- 디자이너 1차안과 함께 평가하면 **방향성 자체**도 검증 가능
- 배포 직전 재평가는 회귀 방지

### 단축안 (예산 제약 시)

**Heuristic + Cultural + Mobile 3명만 + 코디네이터** = 약 600만 원 + 4주
이 조합이 본 프로젝트에서 가장 위험을 많이 잡아냄.

## 7. 주의 사항

1. **편향 방지** — 6명이 서로 사전에 의견 교환하지 않게 격리. Synthesis 워크숍에서만 만남.
2. **코드 접근권** — 평가만이라면 라이브 URL + 영상 캡쳐만으로 충분. 코드 변경 권한 부여 금지.
3. **NDA** — 디자인 시안과 데이터 보호용 비공개 계약 필수 (특히 디자이너 1차안 평가 시).
4. **의뢰자 참여** — Synthesis 워크숍 외에는 평가 과정에 의뢰자가 개입하지 말 것 (편향 발생). 1차 결과 받기 전엔 패널 의견 안 묻기.
5. **결함 vs 취향 구분** — "내 취향과 다르다"가 결함으로 둔갑하지 않게. 코디네이터가 게이트 역할.

**옵션 C. 자체 평가 우선** — Heuristic 평가만 의뢰자가 본 문서 §2 양식대로 직접 시도. 결과 미흡할 시 옵션 B로 escalate

본 의뢰자 입장의 권고: **옵션 C → B → A** 순으로 단계적 검증.

이유: 옵션 C에서 발견한 결함 90%는 셀프로 잡힘. 남은 10%(특히 Cultural / Mobile)가 진짜 외부 시각이 필요한 영역. 옵션 B는 이 10%를 정조준.
