## § 2. 치명/심각 결함 (심각도 3–4)

> 아래 11개 결함은 1차 페르소나(30대 직장인)의 핵심 사용 흐름을 직접 막거나 신뢰를 훼손한다.

### E-002 (심각도 4) — L1 진입 시 "무엇을 하는 앱인지" 30초 내 이해 불가
- 화면 상단에 "조선 왕조 500년" 제목만 있고, 상호작용 가능성·줌 UX 개념·현재 단계를 알려주는 온보딩 힌트가 전무하다.

### E-007 (심각도 4) — 모바일에서 핀치 줌과 OS 페이지 줌 충돌
- `passive: false` + `e.preventDefault()` 조합이 iOS Safari에서 전체 페이지 확대/축소를 차단하지 못하는 경우가 있으며, 차단하더라도 사용자가 의도치 않게 레벨을 넘어갈 수 있다.

### E-014 (심각도 4) — 공유 URL이 level=1일 때 파라미터를 삭제함
- `App.tsx:204–206` — level이 1이면 `window.history.replaceState(null, '', window.location.pathname)`으로 쿼리를 지운다. 결과적으로 L1에서 특정 왕을 보다가 링크를 공유하면 받는 사람은 빈 L1로 진입한다.

### E-001 (심각도 3) — 현재 레벨·위치를 알려주는 단서가 "LV.N" 텍스트 하나뿐
- nav 우상단의 `LV.{Math.floor(level)}` 레이블은 L3.5를 "LV.3"으로 표시해 사용자가 L3.5 존재를 인지하지 못한다. (App.tsx:315)

### E-005 (심각도 3) — L3 → L3.5 진입 여부를 사용자가 예측할 수 없음
- `storyEntry` 유무에 따라 줌인 목적지가 L3.5/L4로 분기하지만, L3 화면에서 이를 미리 알려주는 신호가 없다. "스토리 보기"/"실록 원문" 힌트(Level3DetailView.tsx:116)가 우하단에 `opacity-30`으로 표시되어 발견하기 어렵다.

### E-009 (심각도 3) — TimelineBar 클릭 동작이 레벨에 따라 예측 불가능하게 달라짐
- `selectKingFromTimeline`(App.tsx:149–161): level=1이면 zoomIn, level>2이면 zoomToLevel(2), level=2이면 slideAnimate. 동일 UI 요소가 3가지 동작을 하지만 시각적 구분이 없다.

### E-011 (심각도 3) — L4 원문 없는 사건에서도 실록 뷰로 진입됨
- `Level4SillokView.tsx:14`에서 `currentEvent.sillokEntry ?? kingData.sillokEntry`로 폴백하여 "해당 사건의 원문이 아닙니다" 면책 문구를 노출한다. 사용자는 자신이 찾던 사건의 원문이 아님을 뒤늦게 인지한다.

### E-016 (심각도 3) — SearchModal 키보드 완전 접근성 미구현
- 검색 결과 목록에서 키보드 화살표 키 이동(focus management)이 없다. Tab 키로만 순차 이동해야 하며, 결과가 12개면 Tab 12회 필요.

### E-019 (심각도 3) — L3.5 다크 씬 카드의 나레이션 텍스트 대비율 미달
- `Level35StoryView.tsx:97`: `text-white` on `from-[#0d0d1a]` 배경. 색상 자체는 통과하지만, `opacity` 없이 `text-white`이므로 OK처럼 보이나, 씬 카드 내부 imagePrompt hint(`opacity-20`, Level35StoryView.tsx:103)는 추정 대비율 약 1.3:1로 WCAG AA 기준(4.5:1) 대비 현저히 미달.

### E-021 (심각도 3) — L2 "재위 종료" 구분선이 마지막 왕(순종)에서 빈 문자열 노출
- `Level2EventView.tsx:79`: `kingMeta?.reign?.split('–')[1]?.trim() ?? ''` — 데이터 내 일부 항목이 `–`(en-dash) 대신 `-`(hyphen)을 사용하면 split 실패로 "재위 종료" 옆에 빈 공간만 남는다.

### E-028 (심각도 3) — 뒤로가기(브라우저 Back)가 앱 내 레벨 이동이 아닌 이전 페이지로 이탈
- `App.tsx:203`: `window.history.replaceState`를 사용하여 레벨 변경 시 히스토리 스택을 쌓지 않는다. 사용자가 L4에서 브라우저 Back을 누르면 앱 내 L3으로 돌아오는 대신 이전 사이트로 이탈한다.

### E-003

- **위치**: `src/App.tsx:107–119` — `zoomToLevel` 함수

## § 6. 종합 권고 (5개 핵심 액션)

### 1순위 — 온보딩 힌트 추가 (E-002)
첫 방문자가 30초 안에 "줌인/줌아웃으로 파고드는 앱"임을 이해할 수 있도록 L1에 1줄 힌트 또는 첫 방문 시에만 보이는 인터랙션 유도 요소 추가. 이 앱의 핵심 가치 제안이 진입 시 전달되지 않으면 나머지 모든 UX가 무의미해진다.

### 2순위 — 브라우저 Back 버튼 지원 (E-028)
`replaceState` → `pushState` 전환 및 `popstate` 이벤트 처리. 모바일에서 스와이프 백이 앱을 이탈시키는 것은 1차 페르소나(30대 직장인, 모바일 사용)의 가장 치명적인 마찰점이다.

### 3순위 — L4 폴백 기사 제거, 전용 미완성 상태 화면 구현 (E-011)
데이터 없는 사건에서 무관한 기사를 보여주는 것보다 정직한 "수집 중" 화면이 신뢰도를 높인다. 2차 페르소나(학습자)의 이탈 방지 핵심.

### 4순위 — 모바일 핀치 줌 임계값 조정 및 명시적 줌 UI 제공 (E-007)
핀치 감도를 보수적으로 조정(임계값 30%)하고, 모바일 전용 floating 줌 버튼 또는 명확한 탭 영역으로 의도치 않은 레벨 전환 최소화. 모바일 지하철 시나리오의 가장 직접적인 개선.

### 5순위 — 공유 버튼 및 URL 동기화 정책 정리 (E-014, E-020)
L1에서도 현재 탐색 상태를 공유 가능하게 하거나, 공유 버튼의 표시 조건을 명확히 정의. 복사 실패 시 오류 피드백 추가. 1차 페르소나의 "공유하고 싶다"는 핵심 행동을 직접 지원.
